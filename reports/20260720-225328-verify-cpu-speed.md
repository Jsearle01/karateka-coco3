### §4  Method — why it is behavioural, and why that is stronger here
**MAME 0.281 exposes no clock to Lua at all.** Probed and logged: `cpu.clock`,
`cpu.configured_clock`, `cpu.unscaled_clock`, `cpu.clock_scale`, `cpu.clockscale` — **all nil**
(`build/logs/clk_probe.txt`); `cpu:total_cycles()` does not exist. No `:sam`/`:gime` device exposes
a clock either. So the speed cannot be *read*; it has to be *demonstrated*.

**The instrument.** `HAL_time_vbl_wait` spins in a two-instruction loop
(`src/hal/coco3-dsk/time.s:120-122`):
```
hal_vbl_spin:  cmpb <hal_frame_lo   ; CMPB direct = 4 cycles
               beq  hal_vbl_spin    ; BEQ taken   = 3 cycles   => 7 cycles/iteration
```
Every cycle the engine is not working is burned here, so a read-tap on `hal_vbl_spin` counting hits
per frame yields `spins × 7 = idle cycles`, i.e. **a hard lower bound on the frame's cycle budget**.
A lower bound is sufficient to *falsify* a clock: exceed a candidate window and that candidate dies.

**The control (A/B/A).** Mid-run, Lua pokes `$FFD8` (SAM speed LO), then later `$FFD9` (HI). Guest
work per frame is unchanged, so phase-matched samples give `total_fast = 14 × (spins_fast −
spins_slow)` — self-calibrating, with no assumed absolute clock.

### §8  Uncertainty flags
- **The disk path is UNTESTED** (§5 AC3). This run had no FDC activity, so it says nothing about
  whether a real disk load switches speed. Do not harden the disk-speed rule on this report.
- **`spins × 7` is a LOWER bound**, not the frame total (it excludes work cycles) — which is why the
  argument is framed as falsification (29,736 > 14,929) plus a differential, rather than a direct
  clock read. The 0.4%/0.8% agreements are corroboration, not the primary claim.
- **The 7-cycle loop count is datasheet arithmetic** (CMPB direct 4 + BEQ taken 3), the one
  non-executed input. If it were wrong, both segments scale together and the **A/B ratio and the
  falsification of 0.89 MHz survive** — only the absolute figure would move.
- **Phase-matching is invalid where work differs between segments**: phases `0D`/`0F` produced
  nonsense (0.007 / 0.158 MHz) because the phase state machine advances differently at half clock.
  Reported openly rather than dropped; the primary bound does not depend on them.
- MAME models the SAM speed bit faithfully **as an emulation**; a hardware-edge claim (e.g. FDC
  survival at 1.78 MHz) is **not** settled by this (coco3 idioms §7).
