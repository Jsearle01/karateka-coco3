### Iteration 7 — Off-by-one in sub-pixel column attribution
Visual gate rejected renders. Jay identified "chroma 1 column too far right" in Logo 2.

Diagnostic (this session): raw pixel coordinate sampling from snap 0083 vs apple2.png.
Ref 1: snap first Blue at X=259 (right sub-pixel of Apple II col 129); apple2 first Blue
at X=184 (col 130 = lc=46). Consistently +1 snap pixel offset across both reference
points. Root cause: MAME's chroma spans right-sub of col N plus left-sub of col N+1;
our render placed chroma at the full block of col N+1, which starts 1 snap pixel to
the right of where MAME's chroma first appears.

### Iteration 8 — Option 2 refactor (causal framing)
Initial fix (Option 3): changed rule to "paint at OFF pixel (col N)". Visually correct
but mis-frames the physics — chroma is generated by the ON run, not the OFF pixel.

Refactored to Option 2: chroma attributed to the leading ON pixel (col N+1, run source),
painted at col N via -1 sub-pixel render offset. Code comment documents the NTSC physics:
gap=1 keeps carrier active, producing chroma spanning right-sub of col N + left-sub of
col N+1; at full-pixel render scale the -1 offset aligns with the visual boundary.

Final position: apple2 first Blue at X=180 (col 129). Correct.

## Methodology lessons

### Automated check tautology
Validation checked "every leading position has chroma" using positions derived from the
same rule — internally consistent, not validated against ground truth. Jay's visual
finding correctly overrode Clyde's confident 109/109 automated passes. Ground-truth
validation must be MAME-pixel-first (find MAME chroma; check render at that position)
not rule-first (predict position; check rule predicts same position).

Pattern candidate: "empirical rule validation must start from captured ground truth,
not derived predictions."

### Reference provenance
Label-based provenance ("diagnostic TRUE") is insufficient. Reference files require:
explicit capture path + MAME configuration + verification that pixel values match
known emulator output format. Tool-generated files should be named explicitly
(e.g., "tool_render_baseline.png") to prevent re-interpretation as MAME captures.

### Visual review authority
Jay's repeated visual finding was correct through 6+ automated-check iterations that
disagreed. The visual-verification protocol's role is not just "catch tool bugs" but
"catch specification gaps and implementation drift that automated checks cannot see."

### Visual-verification protocol promotion candidacy
8 successive exercises on a single wave caught real issues at each iteration. Protocol
strengthened for promotion to 6502-6809-conversion-patterns; should note that color
model completeness requires iteration even when mechanical encoding is correct, and
that automated validation must be grounded in MAME-pixel-first methodology.

## Final color model (as implemented)

4-category rule, empirically validated against snap 0083:

1. OFF pixel: Black (default)
2. Isolated ON + bit7=1 + even screen col: Blue (palette 2)
3. Isolated ON + bit7=1 + odd screen col: Orange (palette 1)
4. Leading of adjacent run + gap==1: chroma attributed to this ON pixel,
   color per screen-col parity of ON pixel, painted at col-1 (-1 sub-pixel offset)
5. All other ON pixels (interior/trailing/leading gap≥2): White (palette 3)

Color rule: screen-col parity = (start_col + local_col) % 2.
- even screen col → Blue (bit7=1) [MAME snap 0083 validated]
- odd screen col → Orange (bit7=1) [MAME snap 0083 validated]

Karateka starting columns: Logo 1 = 119, Logo 2 = 84.
Gap threshold: 113/113 gap=1 chroma, 120/120 gap≥2 White (snap 0083).
